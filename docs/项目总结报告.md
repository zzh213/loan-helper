> 一个从 0 到 1、独立完成并真实上线的智能贷款服务平台
> 在线体验：https://loan-helper-665l.onrender.com
> 编写日期：2026-07 · 用途：实习结项 / 作品集 / 汇报

---

## 一、项目一句话介绍

**「小微贷管家」是给小微企业主和普通人的"贷款翻译官 + 私人军师"。**
用户填写自己的真实经营 / 收入情况，系统就能智能匹配贷款产品、给出最优方案与备选方案，并附上额度 / 利率 / 月供估算、风控评分、补贴政策、材料清单，还能把看不懂的银行黑话翻译成大白话。

一句更凝练的定位：**"我们不放贷、不碰钱，只做小微企业和金融机构之间最难的那一环——让信息不对称的双方精准匹配。"**

---

## 二、为什么做这个产品：用户痛点

小微企业贡献了中国约 60% 的 GDP、80% 的城镇就业，但"融资难、融资贵"是老大难问题。调研中发现，真正卡住用户的往往**不是没有产品，而是三个"不知道"**：

| 痛点 | 用户的真实心声 | 后果 |
| --- | --- | --- |
| **不知道能不能贷、能贷多少** | "面对上百个产品无从下手，又怕申请被拒影响征信" | 不敢试、白白错过 |
| **不知道该找谁贷、会不会被拒** | "银行、担保、助贷信息不透明，怕被坑" | 病急乱投医、被中介收智商税 |
| **看不懂银行专业话术** | "'年化''抵押率''征信''LPR'，每个字都认识，连起来不懂" | 看不懂、不敢签 |

**本质：这是一场普通经营者与金融机构之间的"信息不对称"战。** 强监管、高专业门槛的金融领域，把最需要钱的人挡在了门外。

> **关键洞察**：同类产品都在比"谁的产品库大、算法准"，但用户真正的痛点是**"看不懂"和"不敢试"**。谁能把"专业"翻译成"人话"，谁就赢得信任——而信任才是金融产品的第一入口。
> 这个洞察，决定了产品的灵魂功能是**「说人话版解读」**，而不是又一个冷冰冰的利率计算器。

## 三、产品做了什么：核心功能全景

围绕"破除信息不对称 → 用大白话建立信任 → 手把手帮用户走完融资"这条主线，功能可分为五层：

| 层次 | 功能 | 解决哪个"不知道" |
| --- | --- | --- |
| **看得懂** | 💬 说人话版解读、名词小课堂 | 看不懂话术 |
| **算得清** | 双引擎智能推荐（企业贷 / 个人贷）、8 维风控评分卡、多方案横向对比、可视化分析报告 | 能不能贷、能贷多少 |
| **办得成** | 材料清单、银行成品材料（PDF/Word）、预约信贷经理、申请记录与进度追踪、前置预审 | 找谁贷、怎么办 |
| **拿实惠** | 补贴政策匹配、隐藏贴息（区县 / 园区级）、申报指南 | 别漏掉政府扶持 |
| **敢信任** | 大白话 + 诚实的数据来源标注 + 政府担保增信方案 + 合规免责 | 会不会被坑 |

**配套 / 增强功能：**
- 🤖 **虚拟金融顾问**：可对话答疑，支持语音输入、语音朗读、8 款虚拟形象 + 自定义形象、对话记录持久化
- 🎮 **金融知识闯关游戏**：5 关卡 50 道题，寓教于乐，答题后延伸科普
- 📊 **管理后台 + 数据看板**：埋点统计、六步转化漏斗、功能使用排行、每日趋势、流失预警
- 🏛️ **运营中台**：风控规则可配置、产品库 / 政策库可视化维护、工单、客户经理协作 SOP
- 📱 **PWA 支持**：可安装到手机桌面、离线可用

## 四、制作过程：分阶段演进

整个项目并非一次做完，而是**小步快跑、按优先级迭代**，大致经历了以下阶段：

### 阶段 0：核心引擎（MVP）
搭建 FastAPI 后端 + 原生前端，实现最基本的"填表 → 匹配产品 → 打分排序 → 出方案"闭环。产品库、推荐引擎、风控模型都用**确定性规则**实现。

### 阶段 1：功能广度扩展
- 个人贷款功能（企业 / 个人双引擎）
- 行业细分 + AI 智能识别自定义行业
- 补贴政策库匹配、隐藏贴息（下探到区县 / 园区级）
- 虚拟金融顾问（接入大模型，未配置时降级到内置知识库）
- 金融知识闯关游戏、虚拟形象系统
- 一键导出 PDF / Excel 方案报告

### 阶段 2：合规与专业度
- 金融合规整改：定位为"信息中介"、加风险提示、隐私授权、免责声明
- 数据保鲜：核实并更新到最新真实口径（LPR、担保额度、纳税信用等级等）
- 数据来源标注、征信自查指引

### 阶段 3：账号与数据体系
- 手机号登录、企业信息云端保存、个人中心、多角色管理
- 申请记录管理（状态流转、持久化到 SQLite / Postgres）
- 管理后台（口令保护）、隐私修复（脱敏他人手机号）
- 自动化测试从 5 项扩到 41 项，统一测试入口

### 阶段 4：风控 & 匹配模型迭代（核心竞争力）
- 8 维度风控评分卡标准化、可解释化（每个维度给出加分 / 扣分明细，告诉用户"为什么通过率低"）
- 分层产品匹配（低资质→银税贷 / 信用贷；有设备厂房→抵押贷；大额→组合贷）
- 反欺诈规则（虚假营业额、异常高估值、多头借贷拦截）

### 阶段 5：技术基建与后台管理
- 运营中台：风控规则可配置（改权重不用改代码）、产品库 / 政策库可视化维护、工单、数据导出、审计日志
- 埋点 & 数据看板：全链路转化漏斗、核心指标、流失预警
- 性能与安全：接口限流防刷、CORS 收紧、安全响应头（CSP 等）、敏感数据保护

### 阶段 6：商业化 & 增长
- 商业模式设计（渠道佣金 / 增值服务付费 / 补贴代办服务费）
- 私域转化链路、客户经理协作 SOP
- 运营内容与增长体系（资讯专栏 / 公域落地页 / 留存推送）

### 阶段 7：体验打磨（最近迭代）
- 商用交互标准化：分步表单 + 进度条、实时格式校验、加载态 / 按钮置灰、确认弹窗
- 测算结果可视化：原生 Canvas 图表 + 多方案横向对比卡片 + 纯前端导出 PDF
- 抗冷启动重试、Service Worker 缓存策略优化
- 移动端底部导航栏适配与精简

## 五、遇到的困难与解决方案（重点）

### （一）部署上线类

| # | 困难 | 原因 | 解决方案 | 收获 |
| --- | --- | --- | --- | --- |
| 1 | 临时隧道一关机就失效 | cloudflared 临时隧道依赖本机，本质不是托管 | 转向"部署上云"，找免费云平台长期托管 | 分清"临时演示"与"真正上线" |
| 2 | 云平台一路碰壁 | HuggingFace 注册被网络临时封（403）、Zeabur 需付费 | 逐个试错，最终选完全免费的 **Render**，用 Docker 部署，拿到永久网址 | 技术选型要综合成本 + 可达性 + 长期稳定 |
| 3 | 没有代码托管账号 + 推送卡在登录 | Git 推送需要凭证，HTTPS 要令牌鉴权 | 注册账号建仓库，用**访问令牌**推送，令牌用完即清、远程 URL 保持干净 | 理解 Git 凭证机制与**密钥安全**（绝不把 token 写进代码） |
| 4 | 免费云会"休眠"，首访要等约 50 秒 | 免费资源闲置 15 分钟自动回收 | 演示前先手动"唤醒"；后来做**页面加载静默预唤醒 + 请求自动重试** | 理解免费资源限制并主动规避翻车 |
| 5 | 每次更新步骤繁琐易出错 | 部署链路长、人工操作不可靠 | 编写 `deploy.sh` **一键部署脚本**，自动完成提交 / 推送 / 触发部署 / 轮询到上线 | 把重复劳动自动化，是工程效率意识 |

### （二）产品与交互类

| # | 困难 | 原因 | 解决方案 | 收获 |
| --- | --- | --- | --- | --- |

### （三）技术与代码类

| # | 困难 | 原因 | 解决方案 | 收获 |
| --- | --- | --- | --- | --- |
| 10 | 风控评估框重复渲染两次 | 加折叠功能时未删除首屏原渲染 | 定位重复代码，删掉首屏那份 | 改动要检查副作用 |
| 11 | 合规红线——不能夸大宣传 | 金融产品对宣传措辞极敏感 | 全站改词（"达"→"约"、去绝对化数字），加固定免责声明 | 建立金融场景合规意识 |
| 12 | 该不该全用 AI？核心推荐选了规则 | 金融推荐要稳定、可解释、可复现，大模型有幻觉风险 | 核心推荐用**确定性规则引擎**，只在开放问答接入大模型 | 在合适的地方用合适的技术，不为追热点堆 AI |
| 13 | 用户报"匹配失败，请稍后再试" | Render 冷启动 / 部署切换窗口返回 502/503，前端直接报错 | 加 **fetchWithRetry 指数退避重试** + 页面加载静默 ping 预唤醒 + 优化失败文案 | 免费部署要在前端做韧性兜底 |
| 14 | 可视化柱状图贷款名称重叠 / 截断 | 长中文名挤在窄柱子下 | 柱标签改简短"方案 A/B/C/D" + 图表下方图例映射全称 | 图表可读性也是细节 |
| 15 | 多方案横向对比"缺方案 D" | 对比卡片 `slice(0,3)` 只取 3 个，与柱状图（4 个）口径不一致 | 统一改成取 4 个，标签口径对齐 | 前后口径要一致 |
| 16 | 更新了代码用户却看不到（还是旧的） | Service Worker 用"缓存优先"，老浏览器一直读旧缓存 | SW 改为 **HTML/JS/CSS 网络优先、图片缓存优先**，升级缓存版本 | 理解 PWA 缓存陷阱：更新与离线要平衡 |
| 17 | 预审接口报 500 | `preaudit.py` 引用了已删除的变量（死代码）→ AttributeError | 删除死代码引用 | 删代码要清理所有引用点 |
| 18 | 手机端底部导航按钮错乱 | "登录 / 注册"按钮在移动端仍是 `width:100%` 大块，挤乱整行 | 统一为等宽图标按钮 + 横向滑动兜底；后按需求**隐藏**该按钮 | 响应式要为每个断点单独验证 |
| 19 | 无 Node/npm 环境校验 JS 语法 | 环境受限 | 用 Python 的 esprima、quickjs 校验语法与括号平衡 | 因地制宜找替代工具 |
| 20 | CSP 安全头拦截 CDN 脚本 | 出于安全，`script-src` 限制为 `'self'` | 图表用**原生 Canvas** 手绘；PDF 库 jsPDF/html2canvas **下载自托管**，不放宽 CSP | 安全与功能可以兼得 |

## 六、使用的 AI 工具

### 1. AI 编程助手（贯穿全程的"结对程序员"）
本项目的开发全程借助 **AI 编程助手（GitHub Copilot / 大模型代码智能体）** 完成，它扮演了"结对程序员 + 技术顾问"的角色：
- **代码生成与重构**：从后端 API、推荐 / 风控引擎，到前端页面、图表、PDF 导出，大量代码由 AI 辅助生成再人工审校
- **问题诊断**：如冷启动 502、预审 500、Service Worker 缓存陷阱等 bug，AI 协助快速定位根因
- **工程化**：编写一键部署脚本、自动化测试、安全加固建议
- **技术选型咨询**：如"图表该用 Chart.js 还是原生 Canvas""核心推荐该不该用大模型"等决策讨论

> 这种"人 + AI 协作"的模式，让一个人也能独立完成一个涵盖前端、后端、风控、部署、运营的完整产品——**AI 放大了个人的工程杠杆**。

### 2. 大语言模型（产品内置能力）
- **智谱 GLM-4-Flash / 通义千问**：用于**虚拟金融顾问**的开放式问答（配置 API Key 后启用），未配置时自动降级到内置金融知识库，保证可用性
- **应用克制原则**：**只在开放问答场景用大模型**；核心的贷款推荐、风控打分坚持用**确定性规则引擎**——因为金融决策需要稳定、可解释、可复现、可审计，不能有幻觉

### 3. 浏览器端 AI 能力
- **原生语音识别（Web Speech API）**：虚拟顾问支持普通话语音输入
- **语音合成（TTS）**：可朗读顾问回复
- （留学子项目还用到 **Tesseract.js** 做成绩单 OCR，浏览器本地识别、不上传隐私）

## 七、技术栈与架构

| 层 | 选型 | 说明 |
| --- | --- | --- |
| 后端 | Python + FastAPI + uvicorn | 轻量、异步、自带文档 |
| 前端 | 原生 HTML + CSS + JavaScript（单页） | 零框架依赖，加载快、可控 |
| 数据 | SQLite（本地）/ PostgreSQL（云端） | 双后端，支持云端持久化 |
| 部署 | Docker + Render（免费档） | 永久网址、自动构建 |
| 报表 | reportlab（PDF）、openpyxl（Excel）、前端 jsPDF + html2canvas | 服务端 + 纯前端双通道导出 |
| PWA | manifest + Service Worker | 可安装、离线可用 |
| 大模型 | 智谱 GLM-4-Flash / 通义千问（可选） | 顾问问答，降级有兜底 |

**架构要点：**
- **规则引擎与 AI 分工明确**：确定性推荐 / 风控（可审计） + 大模型问答（体验）
- **安全**：CSP 等安全响应头、接口限流、CORS 收紧、敏感数据脱敏、令牌不入库
- **韧性**：前端重试 + 静默预唤醒，对抗免费云冷启动
- **可运营**：埋点看板 + 后台可配置（改规则不改代码）
