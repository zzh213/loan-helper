# 项目挑战与应对 · 小微贷管家

> 记录「小微贷管家」从 0 到上线过程中真实遇到的问题、原因分析、解决方案与收获。
> 用于实习结项答辩 · 2026.07

---

## 一、部署上线类

### 挑战 1:临时隧道一关机就失效
- **问题**:最初用 cloudflared 临时隧道生成公网链接,想让别人也能访问,但它依赖本机,一关机 / 关终端链接就失效。
- **原因**:临时隧道本质是把本机端口暴露出去,不是真正的托管。
- **解决**:意识到必须「部署上云」,由本机演示转向找免费云平台长期托管。
- **收获**:分清「临时演示」和「真正上线」的本质区别。

### 挑战 2:云平台一路碰壁,才最终落地 Render
- **问题**:HuggingFace 注册页被网络临时封(403);转 Zeabur 又需要付费。
- **原因**:平台的网络可达性、免费额度政策各不相同。
- **解决**:逐个试错,最终选定完全免费的 **Render**,用 Docker 方式部署成功,拿到永久网址。
- **收获**:技术选型要综合「成本 + 可访问性 + 长期稳定」,不能只看第一个方案。

### 挑战 3:没有代码托管账号 + 终端推送卡在登录
- **问题**:部署需要代码托管,但当时没有 GitHub 账号;注册后终端 push 又卡在身份验证。
- **原因**:Git 推送需要凭证,HTTPS 方式要求令牌鉴权。
- **解决**:注册账号建仓库,用**访问令牌**完成推送,并让令牌用完即清、远程 URL 保持干净,不把密钥残留在配置里。
- **收获**:理解了 Git 凭证机制和**密钥安全**——绝不能把 token 写进代码或提交历史。

### 挑战 4:免费云会「休眠」,首次访问要等约 50 秒
- **问题**:Render 免费档闲置 15 分钟就休眠,冷启动慢。
- **原因**:免费资源为节省成本会自动回收闲置实例。
- **解决**:接受这个权衡,并在答辩演示前先手动访问一次「唤醒」服务。
- **收获**:理解免费资源的限制,提前规避演示翻车。

### 挑战 5:每次更新代码步骤繁琐、易出错
- **问题**:每次上线要手动 commit、push、触发部署、等待,步骤多容易漏。
- **原因**:部署链路长,人工操作不可靠。
- **解决**:编写 `deploy.sh` **一键部署脚本**,自动完成提交、推送、触发 Render 部署并轮询到上线。
- **收获**:把重复劳动自动化,是工程效率意识的体现。

## 二、产品与交互类

### 挑战 6:结果页信息过载,一次性铺开吓退用户
- **问题**:匹配结果、风控评估、方案对比、补贴政策全堆在首屏,信息量过大。
- **原因**:早期思路是「展示得越全越好」。
- **解决**:改成**精简版 + 查看更多**折叠;操作按钮也只保留常用 3 个,其余收进「更多功能」。
- **收获**:克制地展示,比展示得全面更重要——这是核心的产品思考。

### 挑战 7:小微企业主看不懂专业金融话术
- **问题**:页面术语太多,用户「看不懂、不敢试」。
- **原因**:内容站在金融从业者视角,而非普通用户视角。
- **解决**:做了**「说人话版解读」+ 名词小课堂**,用大白话把方案讲清楚。
- **收获**:找到了产品的差异化与情感内核——把专业翻译成信任。

### 挑战 8:选完城市后区县列表不全
- **问题**:隐藏贴息功能里,选了北京后,区县下拉只有朝阳、海淀等寥寥几个。
- **原因**:地区数据不完整,下级区县树缺失。
- **解决**:补全 `regions.py` 的区县数据,让每个城市的下级区县完整可选。
- **收获**:数据完整性直接影响可用性,细节不能省。

## 三、技术与代码类

### 挑战 9:风控评估框重复渲染了两次
- **问题**:做折叠功能时,首屏和折叠区各渲染了一份风控框,重复显示。
- **原因**:新增折叠区时未删除原有的首屏渲染。
- **解决**:定位重复代码,删掉首屏那份,只保留折叠区。
- **收获**:改动要检查副作用,发现 bug 顺手修掉。

### 挑战 10:合规红线——不能夸大宣传
- **问题**:早期文案写了「通过率达 XX%」「高出 60%」等绝对化表述,有合规风险。
- **原因**:金融产品对宣传措辞非常敏感。
- **解决**:全站改词(「达」→「约」、去掉绝对化数字),并在底部加**固定免责声明**(不发放贷款、仅供参考)。
- **收获**:建立了金融场景的合规意识。

### 挑战 11:该不该全用 AI?核心推荐选了确定性规则
- **问题**:一开始想全部用大模型做推荐,但金融推荐需要稳定、可解释、可复现。
- **原因**:大模型有幻觉风险,结果不可复现,不利于合规审计。
- **解决**:核心推荐用**确定性规则引擎**(零成本、可审计),只在开放式问答场景接入大模型。
- **收获**:在合适的地方用合适的技术,不为追热点而堆砌 AI。

### 挑战 12:怎么证明产品有用?——不能凭感觉
- **问题**:做了很多功能,却无法量化它们是否真的有价值。
- **原因**:缺少数据度量手段。
- **解决**:亲手搭建**埋点系统 + 数据看板**(六步转化漏斗、功能使用排行、每日趋势),并自动标出流失最严重的环节。
- **收获**:从「做功能」升级到「用数据驱动决策」,这是产品经理最核心的能力。
